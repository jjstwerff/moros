# Moros — Developer Art Guide

Three layers of developer art cover every visual element needed to use the map
editor and renderer without final assets. None of them require a 3-D modelling
tool — items use an existing free SVG library, materials are drawn procedurally
in JavaScript, and 3-D geometry is generated by code from the hex map data.

---

## Purpose and scope

Everything in this document is **developer art** — placeholder assets that
exist solely to make the tools usable during development. They are not game
art and must not be treated as such.

Proper game art requires:
- Deliberate artistic direction and a consistent visual style
- An artist with relevant skills (3-D modelling, texturing, concept art)
- Iteration and review cycles that tools and scripts cannot replicate

The extruded SVG icons, procedurally generated meshes, and AI-tool outputs
described here will look rough. That is acceptable and expected — their only
job is to make items identifiable in the editor and renderer so that level
design and mechanics work can proceed without waiting for final art.

**None of this output should be carried into a released version of the game.**
When final art is ready it replaces these assets file-by-file; the item and
material indices never change, so the swap requires no code changes.

---

## Overview

| Layer | What it covers | How it is produced |
|---|---|---|
| Item icons | 32 px palette buttons in the editor | game-icons.net SVG pack |
| Material swatches | 48 px palette buttons in the editor | Procedural `<canvas>` (JS) |
| 3-D geometry | Scene preview + GLB export for Blender | `moros_render` flat-shading |

## Material Swatches

Material swatches are drawn procedurally in loft inside `materials.loft` using
the `graphics::Canvas` primitives. No image files are needed. The same code
also provides `dev_art_color()`, which the GLB exporter uses for flat-shading.

### Loft implementation (`materials.loft`)

All drawing uses `graphics::Canvas` primitives. Arcs are approximated with
16-segment polylines using `sin`/`cos` from the loft stdlib. Stipple and
grass use a seeded LCG so the pattern is identical across every redraw.

```loft
use graphics

// --- LCG for stable pseudo-random patterns ---

// Advance LCG state. Call repeatedly, pass returned value back as new state.
fn lcg(s: integer) -> integer {
    s * 1664525 + 1013904223
}

// LCG value scaled to [0, range).
fn lcg_range(s: integer, range: integer) -> integer {
    ((s & 0x7FFF) * range) / 32768
}

// --- Arc helper (no native draw_arc exists) ---

// Draw a partial circle arc from angle a_start to a_end (radians),
// approximated as 16 line segments.
fn draw_arc(cv: graphics::Canvas, ax: integer, ay: integer, ar: integer,
            a_start: single, a_end: single, color: integer) {
    segs = 16
    step = (a_end - a_start) / segs as single
    px = ax + (cos(a_start) * ar as single) as integer
    py = ay + (sin(a_start) * ar as single) as integer
    for i in 1..segs + 1 {
        a  = a_start + i as single * step
        nx = ax + (cos(a) * ar as single) as integer
        ny = ay + (sin(a) * ar as single) as integer
        cv.draw_line(px, py, nx, ny, color)
        px = nx
        py = ny
    }
}

// --- Dev-art flat-shading colour ---

// Base RGBA colour for a well-known material index.
// Used by both material_swatch() and the GLB exporter.
pub fn dev_art_color(index: integer) -> integer {
    match index {
        1  => graphics::rgb(74,  128, 64),   // grass
        2  => graphics::rgb(107, 68,  32),   // mud
        3  => graphics::rgb(112, 96,  80),   // road
        4  => graphics::rgb(32,  96,  160),  // water
        5  => graphics::rgb(139, 94,  60),   // wood_floor
        6  => graphics::rgb(122, 122, 122),  // stone_tiles
        7  => graphics::rgb(136, 136, 136),  // stone_stair
        8  => graphics::rgb(122, 80,  48),   // wood_stair
        9  => graphics::rgb(139, 48,  48),   // roof_tiles
        10 => graphics::rgb(144, 144, 128),  // gravel
        11 => graphics::rgb(200, 169, 110),  // sand
        12 => graphics::rgb(216, 216, 208),  // marble
        13 => graphics::rgb(122, 96,  64),   // dirt_floor
        14 => graphics::rgb(168, 216, 232),  // ice
        15 => graphics::rgb(160, 128, 48),   // thatch_roof
        16 => graphics::rgb(136, 136, 136),  // spiral_stair
        17 => graphics::rgb(144, 144, 160),  // grand_arc_stair
        _  => graphics::rgb(80,  80,  80),   // unknown fallback
    }
}

// Base RGBA colour for a WallBase string.
pub fn dev_art_wall_color(base: text) -> integer {
    match base {
        "WOOD"        => graphics::rgb(122, 80,  48),
        "HALF_TIMBER" => graphics::rgb(90,  64,  32),
        "BRICKS"      => graphics::rgb(139, 64,  48),
        "STONE"       => graphics::rgb(112, 112, 112),
        "PLASTER"     => graphics::rgb(216, 208, 184),
        "IRON"        => graphics::rgb(80,  80,  96),
        "WATTLE"      => graphics::rgb(160, 144, 64),
        "GLASS_BLOCK" => graphics::rgb(128, 192, 224),
        _             => graphics::rgb(100, 100, 100),
    }
}

// --- Swatch drawing ---

// Draw a w×h material palette swatch for the given well-known index.
// Returns a Canvas with RGBA pixel data (index 0 = transparent checkerboard).
pub fn material_swatch(index: integer, w: integer, h: integer) -> graphics::Canvas {
    overlay = graphics::rgba(0, 0, 0, 64)

    if index == 0 {
        cv    = graphics::canvas(w, h, graphics::rgb(204, 204, 204))
        dark  = graphics::rgb(136, 136, 136)
        sz    = 8
        for ty in 0..h / sz + 1 {
            for tx in 0..w / sz + 1 {
                if (tx + ty) % 2 != 0 {
                    cv.fill_rect(tx * sz, ty * sz, sz, sz, dark)
                }
            }
        }
        return cv
    }

    cv   = graphics::canvas(w, h, dev_art_color(index))
    seed = index * 7919 + 1

    match index {
        1 => {
            // grass: short strokes at randomised angles
            for _ in 0..18 {
                seed = lcg(seed)
                x  = lcg_range(seed, w)
                seed = lcg(seed)
                y  = lcg_range(seed, h)
                seed = lcg(seed)
                dx = lcg_range(seed, 9) - 4
                cv.draw_line(x, y, x + dx, y - 5, overlay)
            }
        }
        2 | 10 | 11 | 13 => {
            // stipple: mud, gravel, sand, dirt_floor
            for _ in 0..40 {
                seed = lcg(seed)
                x = lcg_range(seed, w)
                seed = lcg(seed)
                y = lcg_range(seed, h)
                cv.set_pixel(x, y, overlay)
            }
        }
        3 => {
            // road: horizontal lines
            for row in 0..h / 6 + 1 {
                cv.hline(0, w, row * 6 + 4, overlay)
            }
        }
        4 => {
            // water: 3 sine-wave rows
            wave = graphics::rgba(180, 220, 255, 128)
            for row in 0..3 {
                cy_base = (h / 3) * row + h / 6
                px = 0
                py = cy_base
                for x in 1..w {
                    ny = cy_base + (sin((x as single / w as single) * 9.4248 + row as single) * 4.0) as integer
                    cv.draw_line(px, py, x, ny, wave)
                    px = x
                    py = ny
                }
            }
        }
        5 | 8 => {
            // wood planks / wood stair: grain lines + staggered joints
            for row in 0..h / 10 + 1 {
                cv.hline(0, w, row * 10 + 9, overlay)
                jx = if row % 2 == 0 { w / 2 } else { w / 4 }
                cv.vline(jx, row * 10, row * 10 + 10, overlay)
            }
            if index == 8 {
                cv.draw_line(0, h, w, 0, overlay)
            }
        }
        6 => {
            // stone_tiles: 12 px grid
            for col in 0..w / 12 + 1 { cv.vline(col * 12, 0, h, overlay) }
            for row in 0..h / 12 + 1 { cv.hline(0, w, row * 12, overlay) }
        }
        7 => {
            // stone_stair: grid + diagonal tread
            for row in 0..h / 12 + 1 { cv.hline(0, w, row * 12, overlay) }
            cv.draw_line(0, h, w, 0, overlay)
        }
        9 => {
            // roof_tiles: offset V-arc rows
            for row in 0..5 {
                cols = w / 12 + 2
                for col in 0..cols {
                    ox = if row % 2 == 0 { col * 12 } else { col * 12 - 6 }
                    oy = row * 10
                    cv.draw_line(ox,     oy + 10, ox + 6,  oy,      overlay)
                    cv.draw_line(ox + 6, oy,      ox + 12, oy + 10, overlay)
                }
            }
        }
        12 => {
            // marble: diagonal vein lines
            vein = graphics::rgba(160, 160, 160, 128)
            for _ in 0..4 {
                seed = lcg(seed)
                x0 = lcg_range(seed, w)
                seed = lcg(seed)
                x1 = lcg_range(seed, w)
                cv.draw_line(x0, 0, x1, h, vein)
            }
        }
        14 => {
            // ice: short diagonal crack lines
            crack = graphics::rgba(255, 255, 255, 153)
            for _ in 0..5 {
                seed = lcg(seed)
                x1 = lcg_range(seed, w)
                seed = lcg(seed)
                y1 = lcg_range(seed, h)
                seed = lcg(seed)
                dx = lcg_range(seed, 33) - 16
                seed = lcg(seed)
                dy = lcg_range(seed, 33) - 16
                cv.draw_line(x1, y1, x1 + dx, y1 + dy, crack)
            }
        }
        15 => {
            // thatch_roof: short diagonal stroke pairs
            for _ in 0..20 {
                seed = lcg(seed)
                x = lcg_range(seed, w)
                seed = lcg(seed)
                y = lcg_range(seed, h)
                cv.draw_line(x,     y, x + 6, y - 6, overlay)
                cv.draw_line(x + 3, y, x + 9, y - 6, overlay)
            }
        }
        16 => {
            // spiral_stair: quarter-circle arcs from bottom-left corner
            r = 8
            for _ in 0..w / 8 {
                draw_arc(cv, 0, h, r, -1.5708, 0.0, overlay)  // 0 to π/2
                r += 8
            }
        }
        17 => {
            // grand_arc_stair: centred arc segments showing tread profile
            r = 10
            for _ in 0..w / 10 {
                draw_arc(cv, w / 2, h / 2, r, -2.356, -0.785, overlay)  // 3π/4 wide
                r += 10
            }
        }
        _ => {}  // unknown: solid fill only
    }
    cv
}
```

### Browser output gap

`graphics::Canvas` stores pixels as a `vector<integer>` (packed RGBA). The
package's only PNG export is `save_png(canvas, path)`, which writes to a
filesystem path. The **`--html` build has no filesystem at all** (no args, no
env either — by design), so that call has nowhere to write in a browser.

This is not a missing feature so much as the browser seam: a `--html` program
does its I/O by sending a message to the JS shell via `host_output`, and the
shell replies through `loftPush`. Either option below still works; option B is
the one that needs nothing from upstream.

Two options, in order of preference:

**Option A — add `OpPngBytes`** *(one new native op in the graphics library)*

```loft
// Return PNG-encoded canvas as a base64 text string.
fn OpPngBytes(cv: graphics::Canvas) -> text
#rust"stores.png_base64(&@cv)"

pub fn material_swatch_png(index: integer, w: integer, h: integer) -> text {
    OpPngBytes(material_swatch(index, w, h))
}
```

JS then sets `img.src = 'data:image/png;base64,' + render.call('material_swatch_png', index, 48, 48)`.

**Option B — raw pixel transfer** *(no new native op needed)*

Export the pixel vector as a JSON array. JS unpacks it into `ImageData`:

```loft
// Return pixel data as a JSON array of packed RGBA integers.
pub fn material_swatch_pixels(index: integer, w: integer, h: integer) -> text {
    cv = material_swatch(index, w, h)
    "{cv.data:j}"
}
```

```javascript
const packed = JSON.parse(render.call('material_swatch_pixels', index, 48, 48));
const buf = new Uint8ClampedArray(packed.length * 4);
packed.forEach((p, i) => {
  buf[i*4]   = (p >> 16) & 0xFF;  // R
  buf[i*4+1] = (p >> 8)  & 0xFF;  // G
  buf[i*4+2] =  p        & 0xFF;  // B
  buf[i*4+3] = (p >>> 24) & 0xFF; // A
});
ctx.putImageData(new ImageData(buf, 48, 48), 0, 0);
```

Option B works today with no changes to the graphics library. Option A is
cleaner once `OpPngBytes` is added. Use Option B until then.

## 3-D Geometry — GLB Export

The `moros_render` loft package includes a `map_export_glb()` function that
generates the full scene geometry and writes it as a GLB 2.0 file. This
validates all procedural geometry (hex surfaces, slopes, walls, items) without
needing the live WebGL pipeline.

### What the export produces

One mesh per well-known material index, each with a flat-shaded PBR material
using the colours below. The output is a single `.glb` file readable by
Blender, any glTF viewer, and Three.js.

### Build target

The GLB exporter compiles as a **native binary** — the browser build has no
filesystem, so file writing is a native-target capability — so it can write
files directly. Add to the Makefile:

```makefile
LOFT := ../loft/target/release/loft

# Build the native GLB export tool
glb-tool:
	$(LOFT) --native lib/moros_render/src/moros_glb_tool.loft \
	        -o bin/moros_glb

# Export a specific map file to GLB (MAP=path/to/map.json)
glb:
	bin/moros_glb $(MAP) html/scene-preview.glb
	@echo "Wrote html/scene-preview.glb — open in Blender or a GLB viewer"
```

## 3-D Item Icons — SVG Extrusion

Item icons from the game-icons.net pack can be extruded into 3D meshes for use
as placeholder geometry in the scene renderer and GLB export. This replaces the
missing `draw_item()` cases for furniture, containers, lights, etc. with
recognisable extruded silhouettes instead of nothing.

### Pipeline overview

| Step | Tool | Input → Output |
|---|---|---|
| 1. Download | `curl` | game-icons.net zip → flat SVG files |
| 2. Parse paths | Python `xml.etree` + `svgpathtools` | SVG `d=` attribute → 2D curves |
| 3. Discretise curves | `svgpathtools` | Bezier/arc segments → polyline points (≤ 16 pts per curve) |
| 4. Triangulate | `shapely` + `mapbox_earcut` | Polygon with holes → triangle indices |
| 5. Extrude | Python (manual) | Front face + back face + side walls |
| 6. Export | `trimesh` | Mesh → GLB 2.0 |

### Source SVGs

Download the full icon pack (white on transparent, 512×512 viewBox):

```
https://game-icons.net/archives/svg/zip/ffffff/transparent/game-icons.net.svg.zip
```

Extract and place the flat folder of `.svg` files in `html/icons/items/` as
described in the Item Icons section above. The extrusion script reads from that
same folder.

### Build script

`tools/svg_to_3d.py` — regenerates all item GLBs from the SVG source files.

```
pip install trimesh svgpathtools shapely mapbox-earcut
python3 tools/svg_to_3d.py
```

Output: one GLB per item in `html/icons/items-3d/{icon-name}.glb`.

### Geometry details

- **Scale**: SVG 512×512 viewBox mapped to 0.6 m × 0.6 m in world space,
  centred on the hex centre, sitting on the hex surface (`y = floor_y`).
- **Depth**: 0.1 m extrusion (one side of a thin plaque).
- **Normals**: flat-shaded; front face normal = +Y (up), back = −Y, sides
  outward.
- **Material**: inherits the `item_def.surface` colour (same as other
  `draw_item` geometry), or white if not specified.
- **Holes**: winding-order detection via `shapely`; counter-clockwise
  sub-paths are treated as holes and subtracted before triangulation.
- **Fallback**: if triangulation fails for a complex path, a flat disc
  (radius 0.3 m, 16 segments) is emitted instead so the palette cell is
  never empty.

### Integration with `draw_item()`

Add a `DEFAULT` case to the `draw_item()` switch that loads the pre-built GLB
by icon name and inlines its mesh at the item position:

```
SWITCH item_def.kind:
  PILLAR:  draw_pillar_geometry(...)
  TREE:    draw_tree_geometry(...)
  LADDER:  draw_ladder_geometry(...)
  NEWEL:   emit_cylinder_post(...)
  ARC_PIVOT: RETURN
  DEFAULT:
    // All furniture, containers, lights, etc.
    mesh = load_item_glb(item_def.icon_name)   // cached after first load
    emit_mesh(mesh, wx, floor_y, wz, angle_rad, scale=0.6)
```

### Risks and mitigations

| Risk | Mitigation |
|---|---|
| Complex compound paths fail triangulation | Emit fallback disc; log the icon name |
| Many Bezier curves → too many vertices | Cap at 16 sample points per curve segment |
| Icon filename not found in zip | Editor shows grey question-mark box (existing behaviour) |
| Depth looks wrong for very thin icon shapes | Increase depth constant or add a bounding-box base slab |

## 3-D Item Icons — Claude-Enhanced Procedural Geometry

For items where the SVG extrusion silhouette is too flat to be useful, Claude
can write per-item procedural geometry functions that compose primitive shapes
(boxes, cylinders, cones, lathe profiles) into recognisable 3D props.

### Approach

Claude writes a `tools/item_geometry.py` module with one `build_<item>()`
function per item, returning a `trimesh.Trimesh`. The existing `svg_to_3d.py`
script calls these functions when available and falls back to SVG extrusion
otherwise. No runtime AI call is needed — the functions are static generated
code.

### Per-category feasibility

| Category | Items | Quality | Technique |
|---|---|---|---|
| **Containers** | barrel, chest, crate, sack, amphora, cauldron, jar | Excellent | Lathe profiles (barrel, amphora, cauldron); box + lid (chest, crate) |
| **Lights** | torch, candle, lantern, campfire, brazier, hearth | Excellent | Cylinder + cone flame; box + arch (hearth); bowl (brazier) |
| **Furniture — storage** | wardrobe, bookshelf, desk, workbench, bed | Good | Box compositions; shelves as thin slabs |
| **Furniture — seating** | chair, throne, bench, stool | Good | Box seat + back slab + cylinder legs |
| **Furniture — tables** | round table, stone table, long table | Good | Disc or box top + legs |
| **Structural** | pillar, ladder, newel, tree | N/A | Already handled by existing `draw_item()` cases |
| **Mechanisms** | anvil, well, forge, grindstone, loom, cart | Good | Recognisable geometric approximations |
| **Decorative — simple** | gravestone, obelisk, altar, signpost, notice board | Good | Box / slab / wedge shapes |
| **Decorative — complex** | statue_person, statue_animal, fountain, throne | Poor | Too organic — keep SVG extrusion silhouette |
| **Moros-specific** | ritual circle, beehive, tanning rack, fishing net | Mixed | Beehive (lathe); ritual circle (flat disc); others left as extrusion |

Roughly **55 of 71 items** are well-served by procedural geometry. The
remaining ~16 (organic statues, complex mechanisms like loom and forge) are
better left as extruded silhouettes.

### Geometry constants

Same coordinate system as the extrusion pipeline:

- SVG 512×512 viewBox → 0.6 m × 0.6 m world footprint, centred on hex.
- Base sits at `y = floor_y`; tallest props reach ~1.0–1.5 m.
- Flat-shaded; material colour from `item_def.surface` or a sensible default.

### Workflow

1. Claude writes `tools/item_geometry.py`, grouped by category (containers
   first — most uniform — then lights, then furniture).
2. Each function is checked in the GLB viewer after writing; geometry written
   without live feedback will need a visual review pass.
3. `svg_to_3d.py` imports `item_geometry` and uses it in preference to the
   SVG path when a matching function exists.

### Limitations

- **No organic detail**: barrel and chair work well; a realistic human statue
  or a detailed candelabra with arms do not.
- **One-time cost**: once written the functions are static — no AI dependency
  at build or runtime.
- **Visual review required**: vertex coordinates must be verified in a GLB
  viewer before committing.

## 3-D Item Icons — AI Tool Generation (Organic Shapes)

For the ~16 items that are too organic or mechanically complex for procedural
geometry, an AI text/image-to-3D tool produces better results than either
extrusion or hand-written code.

### Recommended tools

| Tool | Input | Output | Notes |
|---|---|---|---|
| **Meshy** | Text or image | GLB / OBJ / FBX | Best overall for game props; free tier; GLB output directly |
| **Tripo3D (TripoSG)** | Text or image | GLB / OBJ | High geometric quality; handles organic shapes well |
| **Hyper3D Rodin** | Text or image | GLB / OBJ | Strong at stylised game assets |
| **CSM.ai** | Image | GLB / OBJ | Image-to-3D; use the rendered SVG icon as reference |
| **Luma AI Genie** | Text | GLB | Fast; good for silhouette-based props |

**Meshy** is the recommended starting point: free tier, direct GLB 2.0 output,
and game-appropriate poly counts out of the box.

### Workflow

The game-icons.net SVG for each item doubles as a reference image for
image-to-3D tools:

1. Render the SVG to a PNG (white icon on a mid-grey background).
2. Feed it to Meshy (image mode) with a short text prompt, e.g.
   `"medieval stone fountain, low poly game asset"`.
3. Download the GLB and place it in `html/icons/items-3d/{icon-name}.glb`.

Text-only prompts work equally well for items with self-explanatory names.

## Races and Creatures — AI Tool Generation (Meshy API)

When a Meshy Pro licence is available, the batch script below generates GLB meshes for
all 16 playable races and all defined creatures via the Meshy REST API.

### API workflow

```python
import httpx, time, pathlib

API_KEY = "your_meshy_pro_key"
HEADERS = {"Authorization": f"Bearer {API_KEY}"}

def generate_glb(prompt: str, out_path: pathlib.Path):
    # 1. Submit
    r = httpx.post(
        "https://api.meshy.ai/v2/text-to-3d",
        json={"mode": "preview", "art_style": "game-asset", "prompt": prompt},
        headers=HEADERS,
    )
    task_id = r.json()["result"]

    # 2. Poll
    while True:
        r = httpx.get(f"https://api.meshy.ai/v2/text-to-3d/{task_id}", headers=HEADERS)
        data = r.json()
        if data["status"] == "SUCCEEDED":
            glb_url = data["model_urls"]["glb"]
            break
        time.sleep(5)

    # 3. Download
    out_path.parent.mkdir(parents=True, exist_ok=True)
    out_path.write_bytes(httpx.get(glb_url).content)
```

Output paths:
- Races → `html/icons/races/{race-name}.glb`
- Creatures → `html/icons/creatures/{creature-name}.glb`

### Rigging and walk cycles

Meshy Pro exposes an animation endpoint that takes a generated model task ID and produces a
rigged, animated GLB. Run this as a second pass after the model is confirmed.

**Required animations per character** (developer art minimum):

| Animation | Use |
|---|---|
| `idle` | Standing still in scene preview |
| `walk` | Movement on the hex map |

Meshy's auto-rig handles bipeds (humanoid races, humanoid creatures) and quadrupeds (beasts)
separately. Pass `character_type` accordingly.

```python
def animate_glb(model_task_id: str, character_type: str, out_path: pathlib.Path):
    """
    character_type: "biped" for humanoid races and humanoid creatures,
                    "quadruped" for four-legged beasts.
    """
    r = httpx.post(
        "https://api.meshy.ai/v2/text-to-3d-animations",
        json={
            "original_model_task_id": model_task_id,
            "character_type": character_type,
        },
        headers=HEADERS,
    )
    task_id = r.json()["result"]

    while True:
        r = httpx.get(
            f"https://api.meshy.ai/v2/text-to-3d-animations/{task_id}",
            headers=HEADERS,
        )
        data = r.json()
        if data["status"] == "SUCCEEDED":
            glb_url = data["model_urls"]["glb"]
            break
        time.sleep(5)

    out_path.write_bytes(httpx.get(glb_url).content)
```

Animated GLBs embed all clips in a single file. The renderer selects the clip by name
(`"idle"`, `"walk"`) when playing back.

Output paths follow the same convention as the static meshes, with an `_animated` suffix
to keep both versions on disk during review:
- `html/icons/races/{race-name}_animated.glb`
- `html/icons/creatures/{creature-name}_animated.glb`

**Character type reference**

| Type | Applies to |
|---|---|
| `biped` | All 16 races; Bandit, Town Guard, Stitchling, Memory Shade, The Unfinished, Sphinx, Chimera (humanoid torso), Typhon |
| `quadruped` | Wolf, Brown Bear, Giant Boar, Giant Spider, Giant Rat, Giant Serpent, Stag, Lynx, Wolverine, Mole, Mountain Goat, Weasel, Horse, Donkey, Dog, Graftbeast, Jackalope, Cerberus |
| `biped` (winged) | Eagle, Raven, Heron, Falcon, Hippogriff, Griffin, Wyvern, Cockatrice, all Sprites |
| `biped` (elemental) | All Elementals, Colossus variants, Bee Swarm, Vine Crawler, Wood Elemental, Ancient Treant, Iron Shard, The Accumulator, The Split, Screamer |

Verify each result in a GLB viewer — auto-rigs on non-standard body plans (multi-headed,
swarm, elemental) will need manual cleanup before use in final art.
